# v0.1.10

## 概览

**v0.1.10 修复两个同源的「CLI 静默接受非法输入」缺陷：(1) override 吞 typo；(2) `--mode` 绕过 schema 校验。两者都属于「该校验的 CLI 输入被悄悄接受、埋雷到后续」这一族，均由作者用 pin 测试钉住并留有「如改请翻转并记档」标记，本版按标记翻转。**

**(1) override 吞 typo：** 拼错键名的普通 override（如 `train.max_steps=3`，而 recipe 实际键是 `trainer.max_steps`）此前被悄悄接受、随后被 `extra='allow'` 的 Pydantic 校验丢弃，导致覆盖从未生效且无任何报错。现在普通 `a.b=c` override 要求键已存在，缺失即报错并提示用 `++` 显式新增。

**(2) `--mode` 绕过校验：** `lighttrain train --mode bogus` 直接把 CLI 值赋给已校验的 RootConfig（`cfg.mode = mode`），而 Pydantic 默认不在赋值时重校验（`validate_assignment` 关闭），于是 `Literal["lab","prod"]` 被绕过、`mode: bogus` 被写进 run 的配置快照——该快照**过不了自己的 schema**，reload/resume 时才炸。现在 `--mode` 在赋值前对照 schema 允许值校验，非法即报错退出 1。注：所有消费方都按 `mode == "lab"` vs else 分支，bogus 与 prod 行为完全等价，故「保留非标准 mode」的旧理由是空的，无能力损失。

根因是 `_apply_overrides` 在未启用 struct 约束的 DictConfig 上调用 `OmegaConf.update(..., force_add=False)`，OmegaConf 仍会创建缺失的中间键/叶子；`++` 前缀因此形同虚设。本版让普通 override 在写入前校验键路径是否存在，恢复 Hydra 的「set vs add」语义。该行为此前由 `test_pin_apply_override_regular_creates_intermediates_on_unstructured_cfg` 钉死并留有迁移说明，本版按说明翻转。

- **BREAKING (1)**：普通 `a.b=c` override 命中不存在的键现在抛 `ConfigError`（旧版静默新增并被忽略）。迁移：确认键名拼写；若确需新增键，改用 `++a.b=c`（force-add 语义不变）。
- **BREAKING (2)**：`--mode <非 lab/prod>` 现在报错退出（旧版静默接受）。迁移：仅用 `lab` 或 `prod`。允许值从 `RootConfig.mode` 的 `Literal` 推导，schema 改动会自动同步。
- 框架内部所有以编程方式构造的 override（realtime-control degrade patch、`profile` / `frozen-step` 用的步数覆盖）本就使用 `++` 前缀，**不受影响**。
- `extra='allow'` 仍保留：插件仍可携带自有配置键；只是「设置」一个不存在的键现在必须显式 `++`。
- 影响面已核：除 `train --mode` 外，所有 `setup_run_from_config` 调用方都传 `mode=None`（不触发校验）；选用显式校验而非 `validate_assignment=True`，避免 runtime 大量 `cfg.x=...` 赋值被牵连重校验。
- **非破坏（仅文案）**：invariant DSL（`evaluate_check`）的 docstring 不再自称 "safe / sandbox"。代码行为不变（仍挡住 `().__class__.__subclasses__()` 等通用逃逸 + dunder + builtins），但现明确说明它**不是面向不可信输入的安全边界**——整个 `torch` 模块被绑定，`torch.load`（pickle→RCE）/`torch.save` 可达，故仅假设 `check:` 来自可信 recipe 作者（与 `user_modules` 同等信任）。安全审计结论：在 lighttrain 的单人/lab 威胁模型下 operator 本就有完全代码执行权，此非提权；仅修正会误导多租户场景的措辞。`realtime_control` 的 `inject.py`（默认 `allow_inject=True`）同属 by-design lab 工具，本版**不改默认值**，仅记录评估结论。

## 🐞 修复 —— CLI override typo 不再被静默忽略

**症状：** `lighttrain train -c recipe.yaml train.max_steps=3` 跑满了 recipe 原本的 2000 步，且无任何提示——因为 recipe 的键是 `trainer.max_steps`，`train.max_steps` 被当作新键悄悄写入后又被丢弃。

**修复后：**

```text
$ lighttrain dry-run -c recipes/pretrain_causal.yaml train.max_steps=3
config error: Override 'train.max_steps=3': key 'train.max_steps' does not exist
in the config (likely a typo). Use '++train.max_steps=3' to add a new key.

$ lighttrain train -c recipes/pretrain_causal.yaml trainer.max_steps=3   # 正确键，真正生效
```

## 🐞 修复 —— `--mode` 非法值不再绕过 schema

**症状：** `lighttrain train --mode bogus --print-config` 退出 0 并打印 `mode: bogus`，尽管 `RootConfig.mode` 是 `Literal["lab","prod"]`。该非法值会进入 run 的配置快照，reload/resume 时才在校验处报错。

**修复后：**

```text
$ lighttrain train -c recipes/pretrain_causal.yaml --mode bogus --print-config
config error: --mode 'bogus' is not a valid mode; choose one of ['lab', 'prod'].

$ lighttrain train -c recipes/pretrain_causal.yaml --mode prod --print-config   # 合法值照常生效
```
