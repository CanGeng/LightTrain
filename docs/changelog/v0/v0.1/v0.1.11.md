## 概览

**v0.1.11 从根因上消除三次论文复现实验（mamba3 / GaLore / packing）反复撞上的 `user_modules`
导入 seam**，并修掉 packing 实验暴露的其余 3 个 issue。过去三次都是点修单入口/单拼写，
下个入口照犯；本版攻两个结构性根因：

  `load_config`。所有吃 recipe 的命令都经它 → `prep` 家族（`build_prep_runner`）自动获得导入

- **根因 B —「同一文件能有两个模块身份」**：`register()` 改为按**内容身份**幂等。
  `user_modules`（文件 stem 加载）与 `_target_`（点号路径加载）产生两个模块对象、同名重复注册
  时静默 no-op（结构性消除 ISSUE-2）；不同源文件/qualname/行号仍 raise，`force=True` 不变。
  用户可删所有 `force=True`。
- **核心 BREAKING 变更**：内置 `pack` 节点默认策略由隐式 `next_fit` 改为 `concat_chunk`。

零回归：**1963 → 1993 passed**（新增 30 测试，无既有用例失败）。

## BREAKING — 内置 `pack` 默认策略 `next_fit` → `concat_chunk`

### 行为

`pack` 节点新增 `strategy: concat_chunk | next_fit | best_fit`：

- **`concat_chunk`（新默认）** — padding-free 基线：拼接每个文档（+EOS）成一条流，按 `seq_len`
  切块；跨界文档被截断。这是「打包」最常见的含义，也是多数 baseline 报告的口径。
- **`next_fit`（旧的隐式默认，保留、非默认）** — greedy-pad-flush：缓冲溢出时把短缓冲 padding
  到 `seq_len` 后 flush，只切超长文档。streaming 低内存场景仍合法，但浪费 padding。
- **`best_fit`（opt-in）** — 论文 *Fewer Truncations Improve Language Modeling*（Ding et al.,
  ICML 2024）的 Best-Fit-Decreasing；组合算法，本仓仅 toy-scale 验证过，需显式开启。

三策略均吐标准化 `extras`：`truncation_rate / token_utilization / n_truncated_docs /
n_sequences`（旧版仅 `row_count`）。

### 影响 / 迁移

- 仍用内置 `pack` 的 recipe（`recipes/sft_chat.yaml`、`recipes/sft_chat_hf.yaml`）打包输出
  改变 → 节点 fingerprint 变 → 下次 `prep` 重算（正常）。
- 想保持旧行为：显式写 `strategy: next_fit`。

## 新功能

- **`prep-status -c <cfg> --extras`** — 把节点作者放在 `NodeResult.extras` 的领域指标
  （`truncation_rate` 等，已落盘在 `MANIFEST_COMPLETE.json`）渲染成每节点一行，无需手开 manifest。
- **`load` 节点 `source: lines:<path.txt>`** — 一行一文档的纯 `.txt` 语料，每非空行产
  `{"text": <line>}`。

## Bug 修复

- **`prep-clean --orphans` 在多节点图上 `KeyError`** —`cleanup_orphans` 用裸 `_resolve` 循环
  从不填充 `_fp_cache`（`_resolve` 读它取上游 fp），改为调 `self.plan()`。

`kind:`、`lines:` 读 `.txt`、`--extras` 读指标），全链路 `prep → train → parity` 重跑通过，
Level-1 parity 234/234 bins bit-exact。
