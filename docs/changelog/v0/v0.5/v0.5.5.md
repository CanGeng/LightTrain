## 新增 CLI: prune-tokenizer / check-tokenizer

- 设计来源：端口并改进 `/mnt/.../voca-prune` 工具。
- 文件布局：
  - `lighttrain/builtin_plugins/data/prune/` 算法子包：
    `corpus.py` / `algorithm.py` / `langfilter.py` / `save.py` / `remap.py`
    + `__init__.py` 编排。
  - `lighttrain/cli/commands/tokenizer.py`：两个 CLI 函数。
  - `lighttrain/cli/_app.py`：`app.command("prune-tokenizer")(%)` +
    `app.command("check-tokenizer")(%)`。
- 算法核心：lossless bool 集合裁剪。
  1. 扫 corpus（`.json` / `.jsonl` / `.txt` 递归）`encode` 出 seen 集合。
  2. （可选）`--support-lang` 用 langdetect per-token 多语言过滤；special
     token（`<...>` / `[...]`）自动保留。
  3. （可选）`--inherit-ids` 跨批用 `seen_ids.json` 做 set union；fingerprint
     = 旧 vocab_size，跨 base tokenizer 错配会 raise。
  4. Fixpoint worklist 递归闭包：保留 token 的所有 sub-fragment 也加入 seen
     （按字节长度无依赖，O(N·L²) 秒级）。
  5. Special tokens（id ≥ `len(old_bytes_list)`）强制保留（防御性 no-op）。
  6. `mapping_new2old` 按 old_id 升序输出（保 BPE id 压缩；`check-tokenizer`
     据此复现等价）。
- 端口改进（与 voca-prune 的差异）：
  - bool 集合计数替代频次计数（lossless 不需频次）。
  - corpus `.txt` 一行一例（与 `lighttrain init` 默认 `corpus.txt` 一致）。
  - corpus key 统一七种（`text` / `prompt` / `query` / `response` /
    `instruction` / `input` / `output`），去除原 typo `intruction`。
  - 去掉 chatml 模板拼接 —— prune 工具语义不应混入 chat token 约定。
  - `token_mapping.json` + `seen_ids.json` 为 JSON（torch-无关，可人工审查）。
  - `check-tokenizer` failure-first：mismatch>0 `exit(1)`，与 lighttrain
    其他 CLI 一致。
- 输出布局（Q12 决议）：
  - 仅裁 tokenizer：`tokenizer.json` / `tokenizer_config.json` /
    `special_tokens_map.json` / `added_tokens.json`（若原 in）/
    `token_mapping.json` / `seen_ids.json`。
  - 给 `--remap-embed` 时追加：`config.json` / `generation_config.json` /
    切过 `embed_tokens.weight` / `lm_head.weight` 的 `.safetensors`
    （+ `model.safetensors.index.json`）。
- extras：新增 `prune = ["langdetect>=1.0"]`；`--support-lang` 在未装
  langdetect 时响亮报错指明 `pip install 'lighttrain[prune]'`。
- 测试：`tests/cli/commands/test_prune_tokenizer.py` 9 项 — 端到端
  lossless 等价、mapping 单调去重、failure-first、fixpoint 子片段闭包、
  special 强制保留、support_lang zh vs en、缺依赖响亮 raise、CLI 守卫。

## Activation offload 真模式

- 现 `_activation.py` `mode='offload'` 是 `warnings.warn` + fallback
  `recompute`，实为 dead 选项。
- 改动：自定义 `_ActivationOffloadFunction(torch.autograd.Function)`：
  - `forward`：计算 `y`；若 `x.requires_grad` 且 `x.is_cuda`，把
    `x.detach().cpu().pin_memory()` 存到 `ctx.x_pinned`；CPU 输入直接 raise
    `RuntimeError`（fail-loud）。
  - `backward`：`x_pinned.to(device, non_blocking=True)` 预取；
    `enable_grad` 下以 `x.requires_grad_(True)` 重做 layer `forward`，
    用 `y.backward(grad_y_local)` 同时累加参数 grad 到 `.grad` 并拿到
    `x.grad` 返回（标准 activation-checkpoint 模块级 recompute 模式）。
  - 输入 `requires_grad=False` 时跳过 offload 与 bookkeeping（inference 路径
    无额外成本、无 raise）。
- `ActivationManager.wrap` 改：`mode='offload'` 且 `device` 非 CUDA（`None` 或
  `cpu`）时直接 `raise RuntimeError`，避免直接构造能选中却空转的实现
  （experience.md #21 教训的 v0.2.3 死灰复燃）。
- `recompute_or_offload` 行为不变（继续回落 `_CheckpointWrap`）。
- 测试：
  - `tests/layer_offload/test_activation_offload.py::test_offload_equals_recompute_gpu`
    `@pytest.mark.gpu`：真 GPU 上 `offload` vs `recompute` loss & grad 在
    `atol=1e-5` 内对齐。
  - `test_cpu_fail_loud_wrap_raises_runtime_error`、`test_offload_wrap_forward_rejects_cpu_grad_input`、
    `test_offload_wrap_forward_no_grad_input_skips_offload`、`test_recompute_modes_unchanged_on_cpu`。
