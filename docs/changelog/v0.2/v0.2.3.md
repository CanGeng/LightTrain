## 概览

两部分：

1. 承 v0.2.2 的 §3.3 纠偏收尾：把**剩余的两个 objective 实现**（`next_token` / `masked_denoising`）也移入
   [lighttrain/plugins/objectives/](../../lighttrain/plugins/objectives)，并**整包删除** `lighttrain/objectives/`。
   至此 §3.3「objective 具体实现 → frontier（协议留 core）」完全落地：`objective` 类目下**无任何实现留在核心层**。
   （这部分 relocate-only、零功能变化。）

2. **根治实现缺口审计报告**（`IMPLEMENTATION_GAPS_REPORT.md`）的 5 类「配置看似生效、实则被静默忽略」缺口——见下「Gap-report 修复」。SP/EP 一项本轮搁置，登记为「已知问题」。

## 为什么连"基础" objective 也移

v0.2.2 当时把 `next_token`/`masked_denoising` 当作"基础研究默认"留下，但复核后该判断不成立：

- `ObjectiveProfile`（[architectures/profile.py](../../lighttrain/architectures/profile.py)）docstring 明示该抽象是给**非标准** objective 用的；
- 全仓**唯二**用 `objective:` 节点的 recipe 是 `diffusion_eps.yaml` / `jepa.yaml`——都是已下放的 frontier 范式；
- 默认 LM/MLM 训练走 `loss:`（cross_entropy / mlm），**不经 objective 系统**；profile 的 `loss_family="next_token"` 只是字符串标签，不解析 objective 组件；
- 故 `next_token`/`masked_denoising` 注册了却**无任何消费方**，不满足 §3.1 research 判据（"≥2 研究范式共享"），按 §3.3 属 frontier。

## Gap-report 修复

**Finding 1 — `objective:` 成为内部唯一 canonical 训练 seam（默认归 trainer）**
- 新增未注册 core 适配器 `LossOnlyObjective`（[architectures/profile.py](../../lighttrain/architectures/profile.py)）：把 `loss:` 包成 objective（恒等 `prepare_batch` + stamp `loss_family`），使 runtime 只有一个 seam；`CrossEntropyLoss.loss_family = "next_token"`。
- 新增 `RootConfig.objective` 一等字段；顶层 `loss:` 与 `objective:` 互斥（`load_config` 抛 `ConfigError`，引导嵌套写法），合法嵌套 `objective: {loss: ...}` 不误判。
- runtime：`_build_loss` → `_build_objective`（返回 `(objective, source)`，删除万能 CE 注入；删死代码 `_build_loss`）；trainer **构造后** `_wire_objective` 解析默认 + **双向强制**契约。
- **默认归 trainer**：类级声明 `consumes_objective` / `consumes_objective_prepare` / `requires_objective` + `default_objective()`（[trainers/base.py](../../lighttrain/trainers/base.py)）。删除 RL 的 `_recipe_loss_or` 类型嗅探与 preference 的 None 守卫；`reward_model`/`online_distill` 声明为 inline（不消费 seam）。`produce_batch`/`eval` 经单一 `_prepare_with_objective` 路径，`_doc_boundary` 用后剔除不进 model forward。
- **JEPA 真 EMA**：新增 `target_ema` callback（[callbacks/builtins/target_ema.py](../../lighttrain/callbacks/builtins/target_ema.py)）每 optimizer step 调 `model.update_ema()`（与既有 `ema` 的 Polyak 影子区分）；`jepa.yaml` 引用之。

**Finding 2 — `update_rule:` 归位 `engine.update_rule` + 校验**：6 个 recipe 下沉（standard recipe 顺手删掉由 `trainer:` 控制的 `grad_clip`/`accumulate`）；顶层 `update_rule:` 经 `load_config` 抛 `ConfigError`。修 `_RUNTIME_ONLY` 让 `grad_clip` 透传到声明它的 trainer。

**Finding 3 — 缺失 demo 组件经 `user_modules` 注册（demo-only）**：新增 [examples/lab_components.py](../../examples/lab_components.py)（`mlp_toy` / `synthetic_signal`+`signal` / `synthetic_patches`+`patches` / `synthetic_binary`+`x_labels`（one-hot，触发 PCN 监督钳制）/ `sft_jsonl`）+ 夹具 [tests/fixtures/tiny_sft.jsonl](../../tests/fixtures/tiny_sft.jsonl)；5 个 recipe 加 `user_modules:`。`diffusion_eps`/`jepa`/`ff_demo`/`pcn_demo`/`mezo_sft` 现可端到端运行。

**Finding 5 — RWKV：arch_profile 解析 + `_doc_boundary`**：`@register("architecture", "rwkv"|"transformer")` + `_build_arch_profile`（string→`ArchitectureProfile`，未知名报错）；`lighttrain.architectures` 纳入 eager-import。`LineFileTextDataset` 加 opt-in `chunk_size`（首 chunk 标 `_doc_boundary`），`CausalLMCollator` 透传（batch>1 硬报错）；`pretrain_rwkv.yaml` 改 `chunk_size`/`batch_size:1`/`sampler: sequential`，真正按文档重置状态。

**Finding 6 — CLI/文档对齐**：`docs/guide/cli.md` 的 `convert-checkpoint`/`export` flag 对齐实际 Typer 签名；objective seam / arch profile / trainer·engine 作者契约写入 registry/extending 文档。
